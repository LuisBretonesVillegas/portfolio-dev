00

La idea del verano

Enfocar el verano solo en C es la jugada inteligente. Te explico por qué, para que lo hagas con convicción y no con la sensación de estar dejando cosas fuera.

LA TEORÍA SE SIGUE EN CLASE

Procesos, memoria, archivos: son conceptos que entiendes escuchando si tienes la base. No necesitas adelantarlos; necesitas no perderte cuando el profesor los explique con código.

C ES LO QUE CUESTA EN FRÍO

Nadie se atasca en "qué es un proceso". La gente se atasca cuando aparece *ptr, fork() o un semáforo y no sabe leerlo. Ese es el muro que derribas este verano.

C TE SIRVE PARA TODA LA CARRERA

No es solo para SO. Punteros, memoria y estructuras aparecen en EDA, en redes, en compiladores. Invertir el verano en C rinde en media carrera, no en una asignatura.

10 SEMANAS ES HOLGADO

Con diez semanas no vas justo: puedes ir sin prisa, afianzando. Un mini-proyecto por semana hace que no sea leer y olvidar, sino escribir código y que se quede.

El listón que buscas. No es "saber programar en C a nivel profesional". Es: leer un fragmento de C y entender qué hace, escribir un programa pequeño sin bloquearte, y reconocer punteros y structs cuando aparezcan. Con eso, en clase de SO consolidas en vez de descubrir. Ese es exactamente el objetivo, y con 10 semanas lo superas con margen.
01

Debian listo en 5 minutos

Tu dual boot con Debian 13 es el mejor entorno posible para esto: todo lo que necesitas es nativo, incluido lo de POSIX (fork, pthread) que en Windows exigiría montar WSL. Solo instala las herramientas y ya estás.

# Instala compilador, make y el depurador (una sola vez)
sudo apt update
sudo apt install build-essential gdb

# build-essential ya trae gcc. Comprueba que está:
gcc --version

# Compilar un programa y ejecutarlo:
gcc -Wall -Wextra -g hola.c -o hola
./hola
-Wall -Wextra

Activa todos los avisos del compilador. En C esto es oro: te señala errores que en otros lenguajes petarían en ejecución. Úsalo siempre.

-g + gdb

Compila con información de depuración y usa gdb para ejecutar paso a paso cuando algo no cuadre. Aprender a depurar es media asignatura.

Editor

VS Code, Neovim o lo que uses. Con la extensión de C tendrás autocompletado. No te obsesiones con el setup: un editor y la terminal bastan.

valgrind (más adelante)

sudo apt install valgrind. Detecta fugas de memoria y accesos inválidos con punteros. Muy útil a partir de la semana de malloc.

Un consejo de arranque. No inviertas días en configurar el editor perfecto. Con gcc, la terminal y un editor cualquiera ya puedes hacer las 10 semanas enteras. Lo importante es escribir código, compilar, ver el error, corregir. Ese bucle es el que enseña.

Cómo repartir cada semana. No hace falta maratón. Con leer el capítulo del King en 2–3 sesiones y dedicar una tarde al mini-proyecto vas sobrado. Lo importante no son las horas, es la constancia: mejor un rato casi todos los días que un atracón el domingo. El código se aprende con las manos.
Si una semana se atasca, la que se estira es punteros. Las semanas 5–7 (punteros, cadenas, structs) son las duras. Si necesitas dos semanas para punteros en vez de una, tómatelas: es la inversión más rentable de todo el verano. Prefiere entender bien los punteros y recortar la fase POSIX que al revés — POSIX lo verás en clase, punteros los das por supuestos.
03

Punteros sin miedo

Es el concepto que hace que la gente abandone C, y es innecesario. Un puntero es solo una variable que guarda una dirección de memoria en vez de un valor. Nada más. Aquí lo tienes con el mínimo de código para que le pierdas el respeto antes de llegar a la semana 5.

// Una variable normal guarda un valor
int x = 42;

// Un puntero guarda la DIRECCIÓN de otra variable
int *p = &x;     // &x = "la dirección de x"

// Ahora puedes leer x de dos formas:
printf("%d\n", x);   // 42  (directo)
printf("%d\n", *p);  // 42  (*p = "el contenido de lo que apunta p")

// Y modificar x SIN nombrarla, a través del puntero:
*p = 100;
printf("%d\n", x);   // 100  ← ¡x cambió!

Dos símbolos, y siempre significan lo mismo:

SímboloSe leeQué hace
&x"la dirección de x"Te da dónde vive x en memoria. Es lo que guardas en un puntero.
*p"el contenido de p"Va a la dirección que guarda p y te da (o cambia) el valor que hay ahí.
Por qué SO vive de esto. Cuando en clase veas "la MMU traduce una dirección virtual a física", o "el PCB guarda un puntero a la tabla de páginas", estarás viendo punteros. Un puntero es una dirección de memoria; la asignatura entera va de gestionar direcciones. Por eso dominar el cap. 11 del King te desbloquea media teoría de SO sin esfuerzo extra.
El truco para que cuaje: dibújalo. Cada vez que tengas un puntero, haz un dibujo con cajas (variables) y flechas (punteros que apuntan a esas cajas). Los punteros se entienden en papel mucho antes que en la pantalla. Los profesores de SO dibujan exactamente eso en la pizarra.
04

El puente a Sistemas Operativos

Las tres últimas semanas (8–10) tocan la API POSIX: llamadas al sistema de Unix que no son C estándar (el King no las trae) pero que en tu Debian funcionan de fábrica. Es exactamente el código que aparece en tus apuntes de SO. Verlo este verano hace que en clase lo reconozcas en lugar de descubrirlo.

// fork() crea un proceso hijo. Devuelve DOS veces:
//   - al padre le devuelve el PID del hijo
//   - al hijo le devuelve 0
#include <stdio.h>
#include <unistd.h>

int main() {
    pid_t pid = fork();
    if (pid == 0)
        printf("Soy el hijo\n");
    else
        printf("Soy el padre, mi hijo es %d\n", pid);
    return 0;
}

Y el ejemplo estrella de sincronización, que es literalmente el que está en tus apuntes de SO (mutex protegiendo una sección crítica):

// Compilar con: gcc -Wall -pthread contador.c -o contador
#include <pthread.h>

int contador = 0;
pthread_mutex_t lock;

void* incrementar(void* arg) {
    for (int i = 0; i < 1000000; i++) {
        pthread_mutex_lock(&lock);    // entra a la sección crítica
        contador++;                    // ---- SECCIÓN CRÍTICA ----
        pthread_mutex_unlock(&lock);  // sale
    }
    return NULL;
}
El momento "ajá". Ejecuta ese contador primero sin el mutex (quita las dos líneas de lock): verás que el resultado no da 2.000.000, sino un número menor y distinto cada vez. Eso es una condición de carrera, el problema central de la Unidad 2 de SO. Al añadir el mutex, se arregla. Habrás visto con tus ojos lo que en clase te contarán en teoría. Eso es llegar con base.
Qué NO necesitas dominar. No hace falta que memorices toda la API ni que sepas hacer un shell entero. Con haber tocado fork, un par de hilos y un mutex, ya reconocerás el 90% de lo que salga en SO. La profundidad la das en clase; tú vienes a que no te suene a chino.
05

Cinco hábitos que lo cambian todo

Aprender C va menos de leer y más de teclear. Estos hábitos hacen que el verano rinda.

01 · ESCRIBE, NO COPIES

Teclea cada ejemplo tú mismo, aunque lo tengas delante. Copiar-pegar no enseña; teclear y equivocarte, sí. El error de compilación es tu profesor.

02 · COMPILA CON -Wall -Wextra

Siempre. Los avisos del compilador en C te salvan de errores que de otro modo descubrirías tarde y mal. Trata cada warning como un error a resolver.

03 · UN PROYECTO POR SEMANA

No pases de semana sin el mini-proyecto hecho y funcionando. Es lo que convierte "leí sobre punteros" en "sé usar punteros".

04 · DIBUJA LA MEMORIA

Punteros, arrays, structs: dibújalos con cajas y flechas en papel. Lo que no puedas dibujar, no lo has entendido todavía.

05 · ROMPE COSAS A PROPÓSITO

¿Qué pasa si accedes a array[100] de un array de 10? ¿Y si haces *p de un puntero sin inicializar? Romper y ver el fallo enseña cómo funciona por dentro.

06 · GUARDA TU CÓDIGO

Una carpeta por semana, o mejor un repo de git. En septiembre tendrás tu propio material de consulta, escrito por ti, que vale más que cualquier apunte.

Recursos, por si el King se te hace denso. El King es excelente pero extenso. Si una tarde quieres algo más ligero para un concepto suelto, "Beej's Guide to C" es gratis y directo, y para la parte POSIX de las semanas 8–10, "Beej's Guide to Unix IPC" cubre fork, hilos y semáforos con ejemplos claros. Pero no te disperses: el King te basta para todo, los demás son solo apoyo puntual.
En septiembre. Llegarás leyendo C con soltura, con punteros y structs entendidos, y habiendo tocado fork y semáforos. Cuando el profesor de SO abra un ejemplo en C, tú estarás consolidando mientras otros descubren. Justo lo que querías: usar el verano para la base, y dejar la teoría para el curso, donde se entiende mejor con alguien explicándola.
